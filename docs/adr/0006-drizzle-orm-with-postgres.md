# ADR-0006: Drizzle ORM with Postgres

## TL;DR

Drizzle ORM + node-postgres, injected once via a `DB_CONNECTION` DI token from a `@Global() DbModule`. Schema lives in one file (`libs/core/common/src/lib/db/schema.ts`); migrations are generated by Drizzle Kit. Multi-write operations run inside `db.transaction(async (tx) => ...)`. Tests substitute `DB_CONNECTION` with chained `jest.fn()` mocks.

**Status:** Accepted
**Date:** 2026-05-11

## Context

We need a typed query interface to Postgres for a Nest service that values explicit SQL semantics, fast cold starts, and a schema that doubles as TypeScript types. Heavyweight ORMs (TypeORM, Prisma with its generated client) impose an abstraction tax: their query DSL is the source of truth, the schema is generated, the runtime is opinionated, and complex queries fall back to raw SQL anyway.

Requirements:

- Schema-derived types throughout the codebase (`typeof user.$inferSelect`).
- Versioned, reproducible migrations.
- Transactions for multi-write workflows.
- Pessimistic locking when needed (`for('update')`).
- A single DI seam so tests can substitute the DB without monkey-patching.
- No code generation step that contributors have to remember.

Drizzle ORM satisfies these: the schema is TypeScript, the query DSL is a thin layer over SQL, transactions are first-class, and migrations are generated by Drizzle Kit from the schema.

## Decision

The DB layer is **Drizzle ORM** with the **node-postgres** driver, configured once in `libs/core/common/src/lib/db/db.module.ts` (a `@Global()` module).

Wiring:

- The Postgres pool is created from `ConfigService` values (`database.host`, `database.port`, ..., see ADR-0011).
- The Drizzle instance is exposed via the `DB_CONNECTION` DI token: `provide: DB_CONNECTION, useFactory: (configService) => drizzle({ connection: poolConfig })`.
- The module implements `OnApplicationShutdown` and calls `this.db.$client.end()` to close the pool gracefully.

Schema:

- The full schema lives in `libs/core/common/src/lib/db/schema.ts` and is re-exported from the `@<project>/core-common` package barrel. Domain modules import the tables they need from there: `import { user, otp } from '@<project>/core-common'`.
- New tables/columns are added to `schema.ts` and a migration is generated via Drizzle Kit (`apps/core/drizzle.config.ts`) — committed under `apps/core/migrations/`.

Service usage:

- Services inject the connection: `@Inject(DB_CONNECTION) private readonly db: NodePgDatabase`.
- Reads use the query builder: `await this.db.select().from(user).where(eq(user.id, userId))`.
- Multi-write operations MUST run inside a transaction: `await this.db.transaction(async (tx) => { ... })`. Domain events are published inside the transaction so that a failed publish rolls back the writes.
- Row-level locking uses `.for('update')` for operations that need it (e.g., redeeming a single-use token, reserving inventory).
- Mapper helpers (`private mapToWorkspaceMember(...)`) convert raw Drizzle rows into domain DTOs at the service edge.

## Consequences

**Accept:**

- Schema is TypeScript; column additions and type changes are picked up by every consumer at compile time.
- `typeof <table>.$inferSelect` removes the need for parallel row-type definitions.
- Migrations are versioned, reviewed in PRs, and applied via `drizzle-kit migrate`.
- One pool, one shutdown hook, one place to tune connection limits (pool size, idle timeout, statement timeout, connection timeout, keepalive, application name).
- Tests substitute `DB_CONNECTION` with a chained `jest.fn().mockReturnThis()` for unit tests (ADR-0010), or a container-backed Drizzle instance for integration tests.

**Live with:**

- Drizzle's query DSL is less expressive than handwritten SQL for some joins. Complex reporting queries drop to `sql\`...\`` template literals; reviewers should accept this when it improves clarity over forced builder gymnastics.
- Multi-write correctness depends on the author remembering `db.transaction`. There is no automatic transaction-per-request middleware. Reviewers should ask "should this be transactional?" on every multi-write change.
- Publishing a domain event inside a transaction is not a true outbox: if the message broker is unreachable at commit time, delivery is best-effort. The outbox pattern is the long-term solution; plan to add it before relying on at-least-once delivery semantics.
- Drizzle's generated SQL is occasionally not what a hand-written query would produce. Reviewers should EXPLAIN any query in a hot path before assuming the builder did the right thing.
